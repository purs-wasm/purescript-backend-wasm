# 0029. ulib lib distribution and purs-compiler pinning

## Context

The ulib library layer (ADR 0028) ships as a set of PureScript *shadow* sources that must be
compiled to CoreFn (+ externs) and placed where the bin's lib-first resolution finds them
(`<cli>/../lib`). Today `purs-wasm ulib install` builds that lib **on demand**, from the user's
own `.spago/p` with the user's `purs`. That is fine for in-repo development, but it is not a
shippable story:

1. **The lib is purs-version-sensitive.** purs-wasm's CoreFn decoder is verified against a
   specific compiler (`PureScript.CoreFn` — purs **0.15.16**). A lib built with a `purs` whose
   CoreFn format the decoder does not read would mis-decode or fail. So whatever `purs` builds the
   shipped lib is not a free choice.
2. **Building on the user's machine is fragile** — it needs the right `purs` on PATH and the
   package-set sources resolved. A shipped product should carry a **precompiled** lib so a plain
   install just works (ADR 0028's "ship precompiled corefn" note; this is its realisation).
3. **Distribution channel matters** for *where* the lib lives and *whether* a post-install build
   can run at all (nix store path vs `node_modules`; npm's `ignore-scripts`).

ADR 0011 / 0025 cover platform targets and packaging of the *compiled program*; this ADR covers
packaging of the *compiler + its bundled lib*.

## Decision

**Ship a precompiled lib, built with a pinned `purs`, distributed nix-first then npm.**

- **Distribution order: nix first, npm second.** The first supported install is a nix package
  whose store path carries `lib/` precompiled; the bin resolves `<cli>/../lib` there. npm support
  follows, shipping the same precompiled `lib/` inside the package.
- **purs pin = purs-wasm's CoreFn-decoder target (0.15.16), *guarded* by the shadowed packages'
  compiler compatibility — not the minimum.** The shipped CoreFn must be readable by purs-wasm's
  decoder, which fixes the build `purs` to the decoder's target version (0.15.16 today). We do
  **not** pick the minimum compiler: a lib built with an older `purs` would emit CoreFn the decoder
  is not verified against. The registry compatibility data is instead used as a **guard**:
  - For each shadowed package, `spago registry info <pkg> --json` gives `published[<targetVer>].
    compilers` — the `purs` versions that published version supports.
  - The **supported range** for the lib build is the *intersection* of those lists, further
    intersected with the pinned package-set's own compiler constraint
    (`spago.lock` → `workspace.package_set.compiler`, currently `>=0.15.15 <0.16.0`).
  - The build pin (0.15.16) **must be a member of that supported set — bounded on *both* sides.**
    Release tooling (`ulib-compat.mjs`) hard-fails if the pin is **too old** (below the set's
    min — the package set requires a newer `purs`) *or* **too new** (above the newest the shadowed
    packages have been published-tested against — a package hasn't been republished for our `purs`
    yet). Both bounds are recorded in `compat.json` as `pursMin`/`pursMax`; the fix is to re-shadow
    against package versions that support the pin, or to bump the decoder's pin.
- **Two compatibility axes — do not conflate them.** The registry `compilers` list answers
  *source-compile* compatibility ("can `purs` X compile this package's source?"). It says nothing
  about *CoreFn-format* compatibility ("does `purs` X emit CoreFn this backend decodes?"), which is
  a property of **our decoder**, not the registry. So `min(supported range)` (e.g. `0.15.15`) is
  **not** a safe "minimum `purs` for users": if CoreFn changed between, say, 0.15.15 and 0.15.16, a
  user on 0.15.15 would emit CoreFn our 0.15.16-verified decoder mis-reads. Therefore the
  **required `purs` for a user's own project is the decoder's CoreFn-format range, which today is
  exactly the verified version (0.15.16)** — the same as the build pin. We widen it only after
  testing the decoder against another compiler's output, never by reading the registry.
- **The decoder/linker validates `builtWith`.** Every linked module's CoreFn carries `builtWith`
  (the compiler that produced it); the bin checks it against the supported set (`supportedCorefn`,
  currently `["0.15.16"]`) for **both** the user's app modules and the bundled ulib lib, and fails
  loudly on any mismatch. This turns the breaking-CoreFn-change scenario into a clear "rebuild with
  purs 0.15.16" error instead of a silent miscompile — the operational guard behind the axis split
  above.
- **The pin and the compatibility data are recorded in `ulib/compat.json`** (extending ADR 0028's
  record) and regenerated by `ulib-compat.mjs`; the same `--check` guards drift. `pursMin` there is
  the *source-compile* floor (a release diagnostic), distinct from the *CoreFn-format* requirement
  (`supportedCorefn`) the bin enforces.
- **npm `ignore-scripts`: documented, not engineered around.** Because the lib is *precompiled* and
  shipped inside the package, no post-install build is required in the common case. Should a build
  ever be needed, we document the two escapes — add `purs-wasm` to the install-scripts allow-list,
  or run `purs-wasm ulib install` by hand — rather than building machinery to defeat the sandbox.

## Consequences

- A plain install (nix, later npm) yields a working compiler with idiomatic-fast `Data.*` and **no
  network, no purs, no build** at the user's end.
- The release process gains one declared invariant — *the pinned `purs` is a member of every
  shadowed package version's supported-compiler set, neither too old nor too new* — checked by
  `ulib-compat.mjs` at generate/release time (online); `--check` then guards the version data
  offline in CI.
- The pin tracks purs-wasm's decoder, so bumping the decoder to a newer compiler is a deliberate,
  single-point change that the guard re-validates against the package set.
- npm users in an `ignore-scripts` environment are unaffected (precompiled lib) — the documented
  escapes exist only for the rare rebuild case.
- A new shadowed package whose target version drops support for the pinned `purs` is caught at
  release time (guard fails) instead of mis-decoding at a user's build.

## Alternatives considered

- **Pin `purs` to the minimum compiler all shadowed packages support (and/or declare it the user's
  floor).** Rejected: that minimum comes from the *source-compile* axis (the registry `compilers`
  list) and is silent about the *CoreFn-format* axis our decoder actually cares about. If CoreFn
  changed between that minimum and the verified version, both a lib *and* a user's app built at the
  minimum would mis-decode. The required compiler — for the build and for the user — is the
  decoder's verified version (0.15.16), enforced by the `builtWith` guard; the registry minimum is
  only a release diagnostic.
- **Build the lib on the user's machine at install (current `ulib install`).** Kept for in-repo
  dev, rejected as the shipping story: needs `purs` + resolved sources, and breaks under
  `ignore-scripts`. Precompiled-and-shipped is the default.
- **Engineer around npm `ignore-scripts`** (postinstall trampolines, etc.). Rejected as
  unnecessary once the lib ships precompiled, and as fighting a security default.
- **npm first.** Rejected per project direction: nix is the primary toolchain (the CI and dev
  shell are nix), so nix packaging lands first.
